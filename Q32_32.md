## Signed 32.32 — 64-bit fixed‑point

Signed 32.32 is a fixed‑point format using a 64‑bit two’s‑complement integer where the top 32 bits (including sign) represent the integer part and the low 32 bits represent the fractional part. It gives large integer range with very fine fractional resolution — commonly used where very high precision is required while staying in integer arithmetic.

### Layout and scaling
- Total bits: 64 (signed two’s‑complement).
- Format: 1 sign bit + 31 other integer bits + 32 fractional bits.
- Numerical interpretation: treat the 64‑bit integer value N as the real value N / 2^32.

### Range and resolution
- Minimum representable value: −2^31 = −2,147,483,648.0 (represented by 0x8000_0000_0000_0000 → −2^31 when scaled by 2^32).
- Maximum representable value: (2^31 − 1) + (1 − 2^(−32)) = 2,147,483,647.9999999997671694 (0x7FFF_FFFF_FFFF_FFFF).
- Resolution (LSB): 2^(−32) ≈ 2.3283064365386963e−10.

### Common encodings / examples
- 1.0 → 0x00000001_00000000 (decimal 2^32).
- 0.5 → 0x00000000_80000000 (2^31).
- −1.0 → 0xFFFF_FFFF_00000000.
- Smallest positive increment = 1 (raw integer) → 1 / 2^32.

### Conversions
- Integer to 32.32: q = (int64_t)int_val << 32.
- 32.32 to integer (floor toward −∞ via arithmetic shift): int_val = q >> 32.
- Float/double to 32.32: q = (int64_t)llround(f * 4294967296.0) (2^32), then clamp to [INT64_MIN..INT64_MAX] if needed.
- 32.32 to double: f = q / 4294967296.0.
- Note: when converting floats, use double precision for minimal rounding error because fractional resolution is ~2.3e−10.

### Arithmetic
- Addition/subtraction: 64‑bit signed integer arithmetic; overflow wraps — detect and optionally saturate.
- Multiplication:
  - Product of two 32.32 values is conceptually scaled by 2^64 (i.e., Q64). Use 128‑bit intermediate if available: int128_t prod = (int128_t)a * (int128_t)b; result = (int64_t)((prod + (1LL << 31)) >> 32) for rounding. If no native 128‑bit type, implement wide multiply via 64×64 → 128 decomposition.
  - If only 64‑bit intermediate available, you must split operands into high/low 32‑bit parts and compose the 128‑bit result manually.
- Division:
  - To divide A by B (both 32.32) while preserving fractional precision: promote numerator to 96 bits by left‑shifting 32 bits and divide: result = (A << 32) / B (use 128‑bit numerator to avoid overflow).
- Shifts:
  - Left shift multiplies by powers of two (watch overflow).
  - Right arithmetic shift divides by powers of two, preserving sign.
- Multiply–accumulate: use 128‑bit accumulator for sequences of products.

### Rounding and saturation
- Rounding when shifting right: add (1 << (k-1)) before shifting k bits for nearest rounding.
- Saturation: clamp to MIN = 0x8000_0000_0000_0000 and MAX = 0x7FFF_FFFF_FFFF_FFFF when wraparound unacceptable.

### Implementation notes
- Use builtin 128‑bit type (e.g., __int128 on GCC/Clang) for correctness and simplicity:
  - Multiplication example:
    ```c
    int64_t q32_mul(int64_t a, int64_t b){
        __int128 prod = ( (__int128)a ) * ( (__int128)b ); // Q64
        prod += (__int128)1 << 31;                          // rounding
        return (int64_t)(prod >> 32);                      // back to Q32
    }
    ```
  - Division example:
    ```c
    int64_t q32_div(int64_t a, int64_t b){
        if (b == 0) return (a >= 0) ? INT64_MAX : INT64_MIN;
        __int128 num = ( (__int128)a ) << 32;
        return (int64_t)( num / (__int128)b );
    }
    ```
- If __int128 is unavailable, implement 64×64→128 multiply and 128÷64 division via multiword arithmetic (split into high/low 32‑ or 64‑bit parts).

### Use cases and tradeoffs
- Use when you need very large integer range (±2.1e9) and extremely fine fractional precision (~2.3e−10) in deterministic integer-only arithmetic (financial calculations, high‑precision physics, some signal-processing, fixed determinism across platforms).
- Advantages: predictable, deterministic, no floating‑point nondeterminism.
- Limitations: heavier CPU cost (need 128‑bit intermediates or multiword ops), larger memory than 32‑bit formats, careful handling required for multiplication and division to avoid overflow.
