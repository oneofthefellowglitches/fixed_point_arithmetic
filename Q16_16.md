## 16.16 — 32-bit signed fixed-point

16.16 is a 32-bit two’s‑complement fixed‑point format with 16 integer bits (including sign) and 16 fractional bits. It’s widely used in game development for coordinates, velocities, transforms, and other values needing a modest integer range with good fractional precision while avoiding floating‑point.

### Layout and scaling
- Bits: 32 total = 1 sign bit + 15 other integer bits + 16 fractional bits.
- Numerical interpretation: treat the 32‑bit signed integer N as the real value N / 2^16.

### Range and resolution
- Minimum: −2^15 = −32768.0 (represented by 0x80000000 → −32768 * 2^16 / 2^16 = −32768).
- Maximum: (2^15 − 1) + (1 − 2^(−16)) = 32767.99998474121 (0x7FFFFFFF → (2^31−1)/2^16).
- Resolution (LSB): 2^(−16) = 1/65536 ≈ 1.5259e−5.

### Common encodings/examples
- 1.0 → 0x00010000 (65536 decimal).
- 0.5 → 0x00008000 (32768).
- −1.0 → 0xFFFF0000 (−65536).
- 32767.0 → 0x7FFF0000.
- Small fractional example: 0.0000152587890625 = 1 / 65536 → 0x00000001.

### Conversions
- Integer to 16.16: q = int_val << 16.
- 16.16 to integer (floor toward −∞ via arithmetic shift): int_val = q >> 16.
- Float to 16.16: q = (int32_t)roundf(f * 65536.0f), then clamp to [−2^31..2^31−1].
- 16.16 to float: f = q / 65536.0f.

### Arithmetic
- Addition/subtraction: use 32‑bit signed arithmetic. Detect/saturate on overflow if required.
- Multiplication:
  - Multiplying two 16.16 values yields a 64‑bit product scaled by 2^32. To return to 16.16: int64_t prod = (int64_t)a * (int64_t)b; result = (int32_t)((prod + (1LL << 15)) >> 16) — the +1<<15 adds rounding.
  - Use 64‑bit intermediate to avoid overflow and preserve fractional bits.
- Division:
  - To divide A by B (both 16.16) while preserving fractional precision: int64_t tmp = ((int64_t)A << 16); result = (int32_t)(tmp / B). Handle B==0.
- Shifts:
  - Left shift multiplies by powers of two (watch for overflow).
  - Right arithmetic shift divides by powers of two and preserves sign.
- Multiply–accumulate: use 64‑bit accumulator (or widen accumulator) to avoid overflow when summing many products.

### Saturation and overflow
- By default two’s‑complement arithmetic wraps on overflow. For many game uses (positions, physics) you should clamp to the representable min/max when overflow is unacceptable.
- Example clamp bounds: MIN = 0x80000000, MAX = 0x7FFFFFFF.

### Useful idioms (C-style)
```c
// Convert float to 16.16 with clamping
int32_t float_to_16_16(float f){
    int64_t v = (int64_t)llroundf(f * 65536.0f);
    if (v > INT32_MAX) return INT32_MAX;
    if (v < INT32_MIN) return INT32_MIN;
    return (int32_t)v;
}

// Multiply two 16.16 values with rounding
int32_t mul_16_16(int32_t a, int32_t b){
    int64_t prod = (int64_t)a * (int64_t)b;        // Q32
    int32_t res = (int32_t)((prod + (1LL << 15)) >> 16);
    return res;
}

// Divide A by B (both 16.16)
int32_t div_16_16(int32_t a, int32_t b){
    if (b == 0) return (a >= 0) ? INT32_MAX : INT32_MIN; // simple saturating handling
    int64_t tmp = ((int64_t)a << 16);
    return (int32_t)(tmp / b);
}
```

### Practical notes and common uses in game dev
- Good tradeoff: ±32768 integer range with 1/65536 precision — ideal for tile/grid coordinates, fixed physics updates, deterministic lockstep multiplayer.
- Use 64‑bit intermediates for multiply/accumulate-heavy code (physics integrators, matrix ops).
- Choose consistent scaling: store transforms, velocities, time steps with known scaling to avoid repeated conversions.
- Consider using saturation for positions/health to avoid wraparound bugs.
- For performance on platforms without 64‑bit fast multiply, consider Q8.24 or Q2.30 variants if different range/precision tradeoffs better match CPU capabilities.
